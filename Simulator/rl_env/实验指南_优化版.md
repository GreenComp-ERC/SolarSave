```text
2. 更长 timesteps 观察是否收敛
3. 增强回撤惩罚，控制 max_drawdown
4. 增强动作稳定性，控制 mean_action_oscillation
5. 分场景检查是否存在灾难性退化
```

## 3. 优化版环境能力

优化版环境相比初版增加了：

- 多场景随机训练：`--scenario random`
- 增强状态：时间周期、短期供需预测、流动性趋势
- 动作平滑：`--action-smoothing`
- 单步动作限幅：`--max-action-delta`
- 更强回撤惩罚：`--drawdown-penalty`
- 回撤阈值惩罚：`--drawdown-threshold` 与 `--drawdown-threshold-penalty`
- 流动性安全线惩罚：`--liquidity-floor-penalty`
- 分场景 benchmark：输出每个冲击场景下 PPO 和静态 1:3 的对比
- 时间戳输出目录：不同实验不会互相覆盖

## 4. Level 2 必做实验清单

为了完成更可靠的 Level 2，建议至少完成以下实验。

### 4.1 OpenWeather 优化版主实验

目标：用优化版 observation、动作约束和安全惩罚训练一个主模型。

推荐参数：

```text

scenario = random
timesteps = 300000
seed = 7
drawdown_penalty = 2.0
oscillation_penalty = 0.35
unmet_demand_penalty = 0.50
liquidity_floor_penalty = 0.35
drawdown_threshold = 0.35
drawdown_threshold_penalty = 2.0
action_smoothing = 0.20
max_action_delta = 0.08
```

训练命令：

```powershell
python -m rl_env.train_ppo `
  --data-source openweather `
  --scenario random `
  --timesteps 300000 `
  --n-envs 4 `
  --seed 7 `
  --drawdown-penalty 2.0 `
  --oscillation-penalty 0.35 `
  --unmet-demand-penalty 0.50 `
  --liquidity-floor-penalty 0.35 `
  --drawdown-threshold 0.35 `
```

## 5. Level 3 计划展望

Level 2 的 OpenWeather 版本使用的是“真实实时天气快照 + pvlib 物理模拟”，还不是完整历史真实市场数据。Level 3 的目标是把训练数据升级为更接近真实运行的数据。

### 5.1 Level 3 数据目标

Level 3 至少需要三类数据：

```text
1. 历史光伏/天气数据
2. 历史用电需求数据
3. 系统自己的链上市场状态日志
```

建议新增目录：

```text

```

```text
  processed/
```

### 5.2 真实天气/光伏数据来源

优先级建议：

```text
1. Open-Meteo Historical Weather
2. PVGIS
3. NASA POWER
4. NREL NSRDB
```

短期最容易落地的是：

```text

```

```text

```



## 6. 阶段一完成后的模型选择标准

所有阶段一实验完成后，先不要直接接链。建议先选出一个候选模型。

候选模型应满足：

```text
1. mean_liquidity 明显优于 static_1_to_3
2. total_unmet_demand 明显低于 static_1_to_3
3. max_drawdown 不明显差于 static_1_to_3
4. mean_action_oscillation 不过高
5. 分场景没有明显灾难性退化
6. 多 seed 结果稳定
7. 300k 或 1M timesteps 下没有明显过拟合
```

推荐在 `99_reports` 中新增：

```text
phase1_model_selection.md
```

内容包括：

```text
best_model_path
train_args
env_config
benchmark_summary
benchmark_by_scenario
选择理由
已知风险
阶段二接入参数
```

最终模型建议复制或记录为：

```text
rl_env/outputs/99_reports/selected_leader_agent_path.txt
```

不要只凭单次最高 return 选模型，应优先选择稳健模型。

## 7. 阶段二接入计划

阶段二目标是把阶段一选出的 Leader Agent 接入现有链上链下系统，让模型定期输出动态分配参数：

```text
reward_ratio
liquidity_ratio
dynamic_burn_rate
```

### 7.1 合约改造

修改 `smart_contract/contracts/EnergyExchange.sol`。

当前硬编码逻辑：

```solidity
uint256 rewardEnergy = (userEnergy[i] * 25) / 100;
uint256 supplyEnergy = (totalEnergy * 75) / 100;
```

阶段二改为 BPS 动态比例：

```solidity
uint256 public currentRewardRatioBps = 2500;
uint256 public currentLiquidityRatioBps = 7500;
uint256 public currentBurnRateBps = 200;
address public oracle;
```

新增：

```solidity
modifier onlyOracle();
function setOracle(address newOracle) external onlyOwner;
function updateMarketRatios(
    uint256 rewardRatioBps,
    uint256 liquidityRatioBps,
    uint256 burnRateBps
) external onlyOracle;
```

建议事件：

```solidity
event MarketRatiosUpdated(
    uint256 rewardRatioBps,
    uint256 liquidityRatioBps,
    uint256 burnRateBps,
    uint256 timestamp
);
```

安全约束建议：

```text
rewardRatioBps + liquidityRatioBps <= 9800
rewardRatioBps 在合理上下限内
liquidityRatioBps 在合理上下限内
burnRateBps 在合理上下限内
oracle 可由 owner 更换
模型异常时 fallback 到 2500 / 7500
```

### 7.2 部署脚本改造

修改：

```text
smart_contract/scripts/deployAll.js
```

需要：

```text
1. 部署后设置 oracle 地址
2. 初始化 currentRewardRatioBps/currentLiquidityRatioBps/currentBurnRateBps
3. 把 oracle/simulator 私钥写入 Simulator/.env
4. 地址同步到 client/src/utils/contractAddress.json
```

### 7.3 后端 oracle 改造

在 `Simulator/` 下新增或拆分：

```text
Simulator/oracle/
  leader_agent.py
  observation_builder.py
  chain_reader.py
  chain_writer.py
  safety.py
```

后端流程：

```text
1. Web3.py 读取链上状态
2. 读取所有 panels/factories
3. 构造与训练时一致的 observation
4. 加载 selected leader_agent.zip
5. model.predict(observation)
6. 将 action 转成 BPS
7. 经过 safety clamp / fallback
8. 调用 EnergyExchange.updateMarketRatios(...)
9. 调用 updateMarketStep(...)
10. 记录 market_history.csv
```

关键要求：

```text
训练 observation 和线上 inference observation 必须完全一致
action_smoothing/max_action_delta 也必须在线上复现
所有上链参数必须整数化为 BPS
模型加载失败时必须使用静态 2500/7500 fallback
```

### 7.4 前端同步

修改市场展示相关组件，当前仓库中重点可能是：

```text
client/src/components/SolarTrade.jsx
client/src/components/Transactions.jsx
```

### 7.5 阶段二测试顺序

建议顺序：

```text
1. Solidity 单元测试：比例更新、权限、边界值
2. Hardhat 本地部署
3. Simulator oracle dry-run：只推理不发交易
4. Simulator oracle live-run：发 updateMarketRatios
5. market_loop 联动 updateMarketStep
6. 前端读取动态比例
7. 端到端测试：创建 panel/factory -> oracle 更新比例 -> 市场分配 -> 前端显示
```

### 7.6 阶段二验收标准

阶段二最小验收：

```text
1. 合约不再硬编码 25/75
2. oracle 可以定期更新动态比例
3. updateMarketStep 使用当前动态比例
4. 前端可显示当前比例
5. 模型异常时系统能 fallback
6. 每轮 market step 有本地日志
```

## 10. 注意事项

- `openweather` 使用的是真实实时天气快照，不是完整历史天气数据。
- OpenWeather 结果会缓存到 `rl_env/cache/openweather_weather.json`。
- 如果想重新拉取天气，删除缓存文件后重新运行。
- 训练和 benchmark 的环境参数必须保持一致。
- 如果训练时使用 `--legacy-observation`，benchmark 也必须加 `--legacy-observation`。
- 正式实验不要放到 `90_smoke_tests`，统一放到 `03_level2_openweather/experiments` 或 `02_level2_pvlib/experiments`。
